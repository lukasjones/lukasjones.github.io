  • StereoTyping

    Stereotyping

    Typing

    1/10/2015

    Today I'm going to write about stereotype threat. What is stereotype threat you ask? Good question. Stereotype threat is the danger that a negative stereotype can influence the outcome of certain situations in a bad way. For example, If you are in a group that is stereotyped at being worse at science than another group and you do a test with that in mind. Your fear of fulfilling that stereotype can actually make you do worse on the test than you would if there were no stereotype.

    Personally I haven't knowingly felt the affects stereotype threats with regards to academics but I think I've felt them in other aspects of my life. The talk that Claude Steele gave gives an example of white people being stereotyped as racist and I have definitely tried hard not to conform to that stereotype. Sometimes though, in trying to avoid something, you overthink things and end up being the thing you were trying to avoid. The middle school I went to was pretty tough and I was not. There were a lot of fights that broke out and the majority of people fighting were black. Thus I developed an almost overly friendly attitude with black people that I wouldn't have had if I wasn't afraid of the stereotype I had created in my mind. I have mostly overcome my irrational fear nowadays but sometimes I wonder if I'm treating some people better than others.

    My whole life I've tried to treat everyone the same. That was how I was raised and I think for the most part I do a pretty good job. That being said, there are some stereotypes that I listen to. I bike to a climbing gym near an asian mall and I am much more aware of the driving there which has saved me from some pretty serious crashes. I know it's dangerous to judge people without getting to know them first but I don't think that you should just throw out all stereotypes at all times. I guess my rule of thumb is when you are meeting someone for the first time they get a clean slate. No stereotypes, no cover judging just honest getting to know them. However, when you are dealing with people out in the day to day you have to keep some stereotypes in the back of your head so you can avoid potentially dangerous situations. Every time I try to have this conversation with people they poo poo me for validating stereotypes but I think it's important to use common sense and not just throw out everything you have learned in the world.

    So I started this post writing about stereotype threat and it kind of morphed into a conversation about stereotypes.. Oh well. Hope you enjoyed it!

  • Git it?

    Do you Git it?

    version control, git and git hub

    11/20/2014

    Version control is awesome for many reasons. You can work on a project and make commits(save points) along the way that store that version of the project at that time. This is cool because if you get a bug in your program that you don't know how to fix, you can always go back to a previous version of the code that does work. Version control also makes it much easier for many people to work on a project and not overwrite the work of the others. It also makes it really hard to lose your project because there are so many people who have working versions of your project and if your using an online version control website, you also have access to your project there.

    Git helps you track your changes by monitoring the files in a repository which is essentially the folder all your files and subfolders are in. When you make a change to a file, like edit a title of the page or add a new file, it recognizes that you made a change and makes a note of it. When you have made all the changes you want to for the current session, you can then write git status to see what you changed. You can then save (commit) that work so the computer remembers that version of the project at that point in time with the command git commit -m "commit message". You can also use git commit -v which opens the commit message up in a text editor.

    GitHub is an amazing tool to have because it keeps all of your work on a website that you can access from anywhere. This makes collaborating with people super simple. All you have to do is send them the link and they can copy your project and start working on it. When they are done, all they have to do is send a pull request (a request to merge the changes made to the project) which will then be approved or denied by the owner of the project depending on if your contribution is worthy. When you store your code on Github, you can use an open source license to allow other people to use and build off your project which opens a up a lot of potential!

  • What position is that

    What position is that?

    relative, absolute, fixed and.. static

    11/27/2014

    So position is a CSS property that you can assign to an element that affects the location that an element is placed in relation to itself and other elements. There are four options you can choose to set position as: relative, absolute fixed and static.

    Position static is the default value for all elements in css. Basically it means that it will be positioned normally on the page. There are no special qualities to it's position.

    Position relative means that the element is positioned relative to itself. Therefor if you give it an additional property and value of "top: 10px;" you will be positioning it 10px down from where the element is on the page, not in the document. According to css-tricks.com, this also opens up a cool ability to absolutely position elements within the relatively positioned elements and they will stay absolutely positioned relative to the relative element.

    Position absolute means that the position is determined by the top left bottom right values specified in relation to the parent elment. For example if you have a paragraph element in a div element and the paragraph has absolute positioning and top: 30px; and left: 40px;, that element will be 30 pixels from the top and 40 pixels from the left of the inside walls of the div element.

    Fixed positioning means that the element will stay where it is in relation to the viewport. This means that when you scroll down the element stays in a fixed position visible on the screen.

  • Arrays vs Hashes

    Arrays vs Hashes

    Harray!

    12/3/2014

    So, I'm going to start of by introducing what arrays and hashes are exactly. In the broadest sense, arrays and hashes are containers that store data and information. This could be numbers, text, other arrays, other hashes, methods and probably other stuff I can't think of right now. I would say that alot of the time, they are used to model something in real life like a bank account or a person's profile information. Below, I have included an example of an array and a hash.

    # This is an array
    my_array = [1, "two", [3, 4], def count return 5 end] ex.1     # This is a hash
    my_hash = {one: 1, two: 2, three: "three", four: [4, 5]} ex.2

    Arrays and Hashes are both similar and unique. Both have an index that you use to access the contents in the relative object. The way you access these contents is very different however. With an array, you use a numer that corresponds to the information you're trying to access's position in the array (see example 3). It's important to know that when you access an array, the first piece of information is accessed with the number 0, not 1. What you're seeing when you're defining values in the hash, is a key (the first word with a :) and a value (the second piece that could be any of the types of data). When you access a hash, it's similar notation except you aren't using a number. Instead, you are using the key to access the value (see example 4).

    #to access an array
    my_array[1] # => 2
    my_array[2] # => [3, 4] ex.3     # To access a hash
    my_hash[:one] # => 1
    my_hash[:four] # => [4, 5] ex.4

    Another way hashes and arrays differ is in the different ways to add and remove elements from them. Adding to or changing values in arrays and hashes is similar to their way of calling them except that you put an equal sign after and a value(see examples 5 and 6).

    #to access an array
    my_array[1] = "ObiOne" # => "ObiOne"
    my_array[2] # => [3, 4] ex.3     # To access a hash
    my_hash[:one] # => 1
    my_hash[:four] # => [4, 5] ex.4

    It's worth noting that there is a different way to create and access hashes. You can use my_hash = {"one" => 1, "two" => 2} to create the hash and access it by using my_hash["one"] which will return 1.

    Hope this has been helpful!

  • Stay Classy

    Stay Classy California

    All About Classes

    12/16/2014

    Hello blog readers,
    this week, we are going to be talking about classes if you couldn't tell by the title. Classes, like pretty much everything else in Ruby, are objects. They have the same inherent methods as objects and the same properties of an object. The big difference between classes and other ruby objects though, is the ability to create new objects from said classes. Let me explain that a little better.

    Classes are defined by typing class "ClassName" with camel case. You may then choose to define an initialize method that will run every time you create a new object with that class otherwise known as instantiating or creating and instance. In this initialize method, it's common to set some of the variables that you will be using. For example, if you were creating a person object you might want to set an age and name. These variables are set to be something called an instance variable. Instance variables are prefixed with the '@' symbol. The reason we use these is because they can be accessed by any of the methods in a class but won't interfere with other variables outside of the class. After the initialize method, you usually create some other methods that are specific to that class. For example you could create a greeting method that would say "Hello everyone!" when called or an introduction method that said "My name is 'name' and I am '22'". Then all you have left to do is end your methods and class declaration and instantiate a new object from the class. The syntax for that is ClassName.new(variables that go into initialize method). In our example it would be me = Person.new(22, "Lukas"). If you look below, There are examples of everything talked about in this method the way it would look in a ruby file.

    class Person
    def initialize(age, name)
    @age = age
    @name = name
    end

    def greeting
    puts "Hello everyone!"
    end

    def introduction
    puts "My name is #{@name} and I am #{@age} years old"
    end
    end

    me = Person.new(22, "Lukas")
    me.greeting #=> "Hello everyone!"
    me.introduction #=> "My name is Lukas and I am 22 years old"     ×

    Daisy

    1/9/2015

    This week we were given the assignment of researching a specific part of Object Oriented Programming (OOP). I like to choose topics that I understand the least so I can learn the most from it. With that in mind; today we will be talking about class methods.

    A class method is a method that is called directly on the class. For example, if you have a class Car and a class method drive then a class method being called on Car would look like "Car.drive". This is different from instance methods where the notation would be "Car.new.drive".

    Class Methods can be called in a couple different ways which are shown below in the code section.

    # First way to define class method
    class Car
    def self.drive
    puts "class method driving"
    end
    end

    # Second way to define class method
    class Car
    class << self
    def drive
    puts "class method driving"
    end
    end
    end

    # Third way to define class method
    class Car
    end
    new_car = Car.new
    def new_car.drive
    puts "class method driving"
    end

    Class methods are used to perform an action on the class as a whole. There seems to be a good amount of people who believe class methods are unnecessary. At this point, I don't know enough about their reasoning to confirm or deny it. They say that people who rely too heavily on class methods turn their OOP into Procedural Programming. Since we're working in an Object Oriented programming language right now I guess it's better to just stick with instance methods instead of falling into bad habits that will be hard to get out of later.

    Ruby/JavaScript Documentation

    Differences

    1/12/2015

    We started learning JavaScript this week and one of the first things I noticed (coming from Ruby) is the lack of preprogrammed methods in JavaScript. In Ruby there are hundreds of pre programmed methods for arrays and hashes and strings and there are lots of different ways to do different things.     As opposed to Ruby, JavaScript has many methods for manipulating the DOM. If you don't know what Manipulating the DOM is, here is a brief explanation. The DOM stands for the Document Object Model with is a representation of the html elements on your page in a sort of tree like structure. Basically it shows them in a way where you can see which elements are parents and which elements are children. Manipulating the DOM means changing something in the DOM. For example if you had an ul with an id of "list" you could select that ul using the .getElementByID method and change something about it like the inner html or the style applied to it. This creates a really cool dynamic functionality where you can write functions that do things and make those things happen when events happen like a click or a hover. Its cool to be learning it so we can combine these tools on our rails apps.
